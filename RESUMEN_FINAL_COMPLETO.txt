PARTE 3: REQUISITOS DE RAM Y CPU
═══════════════════════════════════════════════════════════════════════════════

🔴 PROBLEMA ENCONTRADO EN ORACLE CLOUD:
"sqlservr: This program requires a machine with at least 2000 megabytes of memory."

CAUSAS:
  1. SQL Server 2022 es un servicio pesado
  2. Requiere MÍNIMO 2 GB RAM dedicada
  3. Oracle Cloud free tier (1 GB) es INSUFICIENTE

✅ REQUISITOS MÍNIMOS POR ENTORNO:

DESARROLLO (Localhost):
  CPU:    2 cores
  RAM:    4 GB (mínimo)
  Disco:  10 GB SSD
  OS:     Windows, macOS, Linux

PRODUCCIÓN (Servidor):
  CPU:    4+ cores (recomendado)
  RAM:    8 GB mínimo, 16 GB+ recomendado
  Disco:  50 GB SSD
  OS:     Linux (Ubuntu 20.04+)

DESGLOSE DE MEMORIA:

  Java Spring Boot:
    └─ Heap:     512 MB - 2 GB (configurable)
    └─ Non-Heap: 200 MB
    └─ Total:    ~1 GB en reposo

  SQL Server (Docker):
    └─ Mínimo:    2 GB
    └─ Recomendado: 4 GB
    └─ Total:     ~4 GB

  SO (Ubuntu):
    └─ Kernel:   300-500 MB
    └─ Sistema:  500-700 MB
    └─ Total:    ~1 GB

  TOTAL RECOMENDADO: 8-12 GB RAM

✅ SOLUCIONES:

OPCIÓN 1: Aumentar RAM en Oracle Cloud
  Instance Type: VM.Standard.E2.1 (2 vCPU + 4 GB RAM)
  Costo: ~USD 9-20/mes
  Status: ✅ Funciona perfectamente

OPCIÓN 2: SQL Server con Límite de Memoria
  MSSQL_MEMORY_LIMIT_MB=768
  RAM Total: 2-2.5 GB
  Costo: Free tier
  Status: ✅ Funciona con limitaciones

OPCIÓN 3: PostgreSQL en lugar de SQL Server
  RAM Requerido: 500 MB - 1 GB
  Costo: Free tier
  Status: ✅ Mucho más ligero
  Cambio: Necesita ajustes en pom.xml

OPCIÓN 4: Usar MariaDB/MySQL
  RAM Requerido: 500 MB - 1 GB
  Costo: Free tier
  Status: ✅ Compatible

PARTE 4: INSTALACIÓN EN ORACLE CLOUD UBUNTU
═══════════════════════════════════════════════════════════════════════════════

🚀 PASO A PASO COMPLETO:

PASO 1: Crear Instancia en Oracle Cloud Console
  1. Compute → Instances → Create Instance
  2. Image: Ubuntu 22.04 LTS
  3. Shape:
     • Recomendado: VM.Standard.E2.1 (2 vCPU, 4 GB RAM)
     • Free tier alternativa: VM.Standard.E2.1.Micro (1 vCPU, 1 GB)
  4. VCN/Subnet: Default
  5. Asignar IP Pública: Sí
  6. Descargar SSH Key (.pem)
  7. Create Instance

PASO 2: Conectarse via SSH
  $ chmod 600 ~/Downloads/ssh-key.key
  $ ssh -i ~/Downloads/ssh-key.key ubuntu@<ORACLE_PUBLIC_IP>

PASO 3: Instalar Prerequisites
  $ sudo apt update && sudo apt upgrade -y
  $ sudo apt install -y openjdk-17-jdk-headless maven git podman

PASO 4: Clonar Proyecto
  $ cd /home/ubuntu
  $ git clone https://github.com/tu-usuario/veritusbot.git
  $ cd veritusbot

PASO 5: Iniciar SQL Server (CON LÍMITE DE MEMORIA SI NECESARIO)

  Para 4 GB RAM (recomendado):
  $ podman run -d \
    --name sqlserver \
    -e "ACCEPT_EULA=Y" \
    -e "MSSQL_SA_PASSWORD=SqlServer!2026#" \
    -e "MSSQL_MEMORY_LIMIT_MB=2048" \
    -p 14333:1433 \
    mcr.microsoft.com/mssql/server:2022-latest

  Para 2 GB RAM (mínimo):
  $ podman run -d \
    --name sqlserver \
    --memory=1.5g \
    -e "ACCEPT_EULA=Y" \
    -e "MSSQL_SA_PASSWORD=SqlServer!2026#" \
    -e "MSSQL_MEMORY_LIMIT_MB=768" \
    -p 14333:1433 \
    mcr.microsoft.com/mssql/server:2022-latest

  Esperar a que inicie:
  $ sleep 45
  $ podman ps | grep sqlserver

PASO 6: Compilar y Ejecutar Aplicación
  $ mvn clean compile
  $ nohup mvn spring-boot:run > veritus.log 2>&1 &

  O con límite de memoria:
  $ nohup java -Xmx1g -Xms512m -jar target/veritusbot-0.0.1-SNAPSHOT.jar > veritus.log 2>&1 &

  Verificar:
  $ ps aux | grep java
  $ tail -f veritus.log

PASO 7: Configurar Firewall (Security Lists)
  En Oracle Cloud Console:
  1. Networking → Virtual Cloud Networks
  2. Seleccionar VCN
  3. Security Lists → Default
  4. Add Ingress Rule:
     • Source CIDR: 0.0.0.0/0 (o tu IP específica)
     • Protocol: TCP
     • Destination Port: 8083
  5. Add

PASO 8: Probar Acceso
  $ curl -X POST http://<ORACLE_PUBLIC_IP>:8083/api/veritus-app/login \
    -H "Content-Type: application/json" \
    -d '{"email":"admin@veritus.com","password":"admin123"}'

  Respuesta esperada: {"token":"eyJ...","usuario":{...}}

PARTE 5: RESOLUCIÓN DE PROBLEMAS
═══════════════════════════════════════════════════════════════════════════════

❌ PROBLEMA: "Out of Memory" en Oracle Cloud
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • Aplicación se detiene sin mensaje
  • SQL Server no inicia
  • Java process termina

Soluciones:
  1. Aumentar RAM: Cambiar a VM.Standard.E2.1 (4 GB)
  2. Reducir Java heap: -Xmx512m
  3. Usar PostgreSQL en lugar de SQL Server
  4. Aplicar limite en SQL Server: MSSQL_MEMORY_LIMIT_MB=768

✅ RECOMENDADO: Opción 1 (aumentar RAM)


❌ PROBLEMA: "Connection refused" en puerto 14333
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • Error: "Unable to connect to database"
  • Login falla

Soluciones:
  1. Verificar SQL Server corriendo:
     $ podman ps | grep sqlserver

  2. Si no está corriendo, iniciar:
     $ podman run -d --name sqlserver ...

  3. Verificar logs:
     $ podman logs sqlserver | tail -20

  4. Verificar firewall:
     $ sudo ufw allow 14333
     $ sudo ufw allow 8083

✅ RECOMENDADO: Ejecutar el comando completo del PASO 5


❌ PROBLEMA: "Port 8083 in use"
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • Error: "Address already in use: 8083"

Soluciones:
  1. Encontrar proceso:
     $ sudo lsof -i :8083

  2. Matar proceso:
     $ sudo kill -9 <PID>

  3. Cambiar puerto temporalmente:
     $ java -Dserver.port=8084 -jar ...

✅ RECOMENDADO: Matar proceso anterior y reiniciar


❌ PROBLEMA: "CORS error" desde frontend
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • "Access to XMLHttpRequest blocked by CORS policy"

Soluciones:
  1. Actualizar app.cors.allowed-origins:
     app.cors.allowed-origins=https://tu-dominio.com

  2. En desarrollo local:
     app.cors.allowed-origins=http://localhost:3000,http://localhost:4200

✅ RECOMENDADO: Configurar en application.properties


❌ PROBLEMA: "Token inválido o expirado"
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • Error: "Token invalid or expired"

Soluciones:
  1. Hacer login nuevamente
  2. Verificar formato: "Authorization: Bearer <token>"
  3. Verificar no haya espacios
  4. Verificar token no expiró (máximo 24h)

✅ RECOMENDADO: Hacer nuevo login


❌ PROBLEMA: "Usuario o contraseña incorrectos"
═════════════════════════════════════════════════════════════════════════════

Síntomas:
  • Error: 401 Unauthorized

Soluciones:
  1. Verificar usuario existe en BD:
     $ podman exec sqlserver sqlcmd -S localhost -U sa -P "SqlServer!2026#" \
       -Q "SELECT * FROM usuarios"

  2. Insertar usuario de prueba:
     (Ver archivo insert_test_usuarios.sql)

  3. Verificar contraseña es correcta

✅ RECOMENDADO: Ejecutar script SQL de prueba

MONITOREO Y MANTENIMIENTO
═══════════════════════════════════════════════════════════════════════════════

Script de Health Check:

#!/bin/bash
# /home/ubuntu/health-check.sh

# Verificar Java
if ! pgrep -f "java" > /dev/null; then
    echo "❌ Java process not running!"
    systemctl restart veritus || exit 1
fi

# Verificar SQL Server
if ! pgrep -f "sqlserver" > /dev/null; then
    echo "❌ SQL Server not running!"
    podman start sqlserver || exit 1
fi

# Verificar endpoint
RESPONSE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8083/api/veritus-app/health-auth)
if [ "$RESPONSE" != "200" ]; then
    echo "❌ API not responding. Status: $RESPONSE"
    exit 1
fi

# Monitorear memoria
MEMORY=$(free -m | awk '/^Mem:/{print $3}')
if [ $MEMORY -gt 3500 ]; then
    echo "⚠️  High memory usage: ${MEMORY}M"
fi

echo "✅ All systems operational - Memory: ${MEMORY}M"

Configurar cron job:
$ crontab -e
*/5 * * * * /home/ubuntu/health-check.sh >> /var/log/veritus-health.log 2>&1
